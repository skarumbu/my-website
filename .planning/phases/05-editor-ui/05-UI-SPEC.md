## Spacing Scale

Declared values (multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, badge inner padding, inline margins |
| sm | 8px | Between badge and date, between label and input |
| md | 16px | Default element spacing, input padding, row gaps |
| lg | 24px | Section padding (matches posts-content padding-top) |
| xl | 32px | Vertical space above headings, toolbar-to-list gap |
| 2xl | 48px | Page top padding (NavBar clearance) |
| 3xl | 64px | Bottom page padding (prevents content hiding behind fixed bars) |

Exceptions:
- Touch target for delete icon button: minimum 44px hit area (use padding to inflate without changing visual size)
- Textarea body field: grows from `min-height: 320px` with no upper bound cap (CSS `resize: vertical`)

**Source:** Inferred from `posts-content` padding (24px/40px) and `ideas-content` (24px/40px). Adopted standard 8-point scale as default.

## Typography

| Role | Family | Size | Weight | Line Height | Usage |
|------|--------|------|--------|-------------|-------|
| Body | DM Sans | 16px | 400 | 1.5 | Description text, post row descriptions, form field values, editor title input, textarea body |
| Label | DM Sans | 13px | 400 | 1.4 | Form field labels, date stamps, status badges, nav items, autosave indicator, error messages, back link |
| Display | Fraunces | 22px | 600 | 1.25 | Post title in list rows (matches `.posts-row-title` exactly) |
| Heading | Fraunces | 28px | 600 | 1.2 | Page heading ("Your posts") on `/write` list |

Notes:
- Editor title input: 16px, weight 600, DM Sans — the title field uses body size at semibold weight to feel editorial without requiring a separate size token
- Editor description input: 16px, weight 400, DM Sans — matches body role
- Textarea body: 16px, weight 400, DM Sans, line-height 1.7 — slightly looser than body for markdown authoring comfort
- Autosave status indicator: 13px, weight 400, DM Sans, color `--ink-3` — dim, unobtrusive (absorbs the former 12px slot)
- The 500 (medium) weight is not used — anything formerly at 500 uses either 400 (regular) or 600 (semibold)

**Source:** Font families from `posts.css` and `ideas.css`. Sizes aligned with existing `.posts-row-title` (22px/Fraunces) and `.posts-row-date` (13px/DM Sans) to keep the list page visually consistent with the public `/posts` page. Consolidated from 7→4 sizes and 3→2 weights per checker requirements.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#FFEFD9` (`--bg` / `--bg-1`) | Page background for both `/write` and editor pages |
| Secondary (30%) | `rgba(255,255,255,0.72)` | Toolbar bar, form field backgrounds, list row hover state |
| Accent (10%) | `#E5533C` (`--accent`) | "Save" button background only; active state ring on "Published" checkbox |
| Destructive | `#B83D2A` (`--accent-dk`) | Delete button text/border; destructive confirm dialog confirm button |
| Good | `#4FA86E` (`--good`) | "Published" badge in list rows; "Published" checkbox checked indicator |

**Accent reserved for:** Save button background, Published checkbox active ring. Not used for any other interactive element.

**Surface distinction — editor vs. candy theme:**
The editor pages do NOT inherit the animated dot-grid (`::after`) or warm radial gradient overlay (`::before`) from `.main`. The write pages use a flat `--bg` background — same warm cream color but without the playground texture. This is the sole visual distinction that makes the editor feel calm and focused. The font, color tokens, and nav are shared.

**Source:** All hex values from `src/styling/ideas.css` `:root`. Design direction from `<design_direction>` in prompt.

## Interaction States

### Save Button
- Default: accent fill `#E5533C`, white text, `border-radius: 8px`
- Hover: `#B83D2A` (darken 15%), cursor pointer
- Active/saving: opacity 0.65, cursor not-allowed, label changes to "Saving…"
- Disabled: not applicable — Save is always available

### Published Checkbox
- Unchecked: native checkbox + "Published" label in `--ink` color
- Checked: native checkbox (browser-default check) + label stays `--ink`; the checked state is the authoritative visual — no custom toggle needed in v1

### Delete Button (list row)
- Default: transparent background, `--ink-3` trash icon
- Hover: icon turns `--accent-dk`; background `rgba(184,61,42,0.08)`

### Delete Post Button (editor toolbar)
- Default: transparent background, `--accent-dk` text, `1px solid --accent-dk` border; `border-radius: 8px`; `padding: 8px 16px`
- Hover: background `rgba(184,61,42,0.08)`

### List Row
- Default: no background; `border-bottom: 1px solid var(--line)` — matches `.posts-row` exactly
- Hover: `background: rgba(255,255,255,0.6)`, `padding-left: 16px` (shift-right animation, 180ms ease) — matches `.posts-row:hover` exactly
- Title is an `<a>` tag linking to editor; delete button is a separate `<button>` inside the row (stops propagation)

### Autosave Indicator
- "Saved" state: `--good` color (#4FA86E), displayed for 3 seconds then fades to invisible
- "Saving…" state: `--ink-3` color
- "Unsaved changes" state: `--ink-3` color; appears after any edit when >30s since last API save
- No indicator shown on initial load (clean state)

### Unsaved-Changes Warning
- `beforeunload` event: browser native dialog (no custom UI)
- React-router navigation block: use `useBlocker` (react-router-dom v6.4+) — show an inline banner above toolbar: "You have unsaved changes." with "Leave anyway" and "Stay" buttons if `useBlocker` is available; fall back to `window.confirm()` otherwise

## Autosave Status Display Logic

```
On any field change:
  → set autosaveStatus = "Unsaved changes"

On localStorage write (every 2-3s):
  → no indicator change

On API save attempt:
  → set autosaveStatus = "Saving…"

On API save success:
  → set autosaveStatus = "Saved"
  → after 3000ms: set autosaveStatus = "" (invisible)

On API save failure:
  → set autosaveStatus = "Unsaved changes"
  → show editor-error message
```

## Accessibility Notes

- All form inputs have visible `<label>` elements (not placeholder-only)
- Delete icon buttons have `aria-label="Delete post"` / `title="Delete post"`
- Confirm dialogs use native `window.confirm()` — inherently accessible (keyboard navigable)

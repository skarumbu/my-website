<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Posts API | Client fetches `/api/posts/:slug`; the post `body` is an untrusted markdown string rendered into the DOM |
| URL → app | `:slug` path param is attacker-controllable and interpolated into the fetch URL |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-04 | Tampering (XSS via markdown body) | PostReader.tsx rendering post.body | mitigate | Render via `<Markdown>{post.body}</Markdown>` (react-markdown uses React.createElement, never emits raw HTML). No `dangerouslySetInnerHTML` (grep gate, Task 1). Do NOT enable `rehypeRaw`. SEC-01 satisfied. |
| T-03-05 | Spoofing (open redirect via markdown link) | links inside post.body | mitigate | react-markdown's default `urlTransform` blocks `javascript:`/`data:` URLs — keep the default (no urlTransform override). |
| T-03-06 | Tampering (slug injection in fetch URL) | useParams slug → fetch | mitigate | slug is only interpolated into a same-origin API GET path; react-router already URL-decodes the param. The API (Phase 2) validates the slug server-side and returns 404 for unknown slugs, surfaced as the inline error state. |
| T-03-07 | Information Disclosure (error detail leak) | error message rendering | mitigate | The reader error copy is a generic user-facing message ("Could not load this post...") — raw `${res.status} ${res.statusText}` is caught into `error` but the rendered fallback uses the generic copy; no stack traces or internal paths exposed (ASVS V7). |
</threat_model>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Bicep → Azure Resource Manager | IaC declares resources; ARM enforces them |
| Azure Functions managed identity → Blob Storage | RBAC-controlled; no connection string in app code |
| GitHub Actions → Azure | Service principal credentials in GitHub secrets |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Information Disclosure | Azure Blob Storage container public access | mitigate | postsapi.bicep sets allowBlobPublicAccess: false on storage account (account-level hard override) AND publicAccess: 'None' on the posts container (container-level). Both are required — belt and suspenders. Acceptance criterion verifies both strings are present in postsapi.bicep. |
| T-01-02 | Information Disclosure | storageConnectionString in Bicep outputs | accept | Output is used for CI/CD wiring only; it is a deployment output, not a logged value. Production access uses managed identity (RBAC), not the connection string. Connection string output is the Azure Functions pattern (same as digitsfunctions.bicep). |
| T-01-03 | Elevation of Privilege | RBAC role assignment with wrong role ID | mitigate | Task 3 checkpoint requires `az role definition list` to verify ba92f5b4-2d11-453d-a403-e96b0029c9fe before deploy. If ID differs, postsapi.bicep is corrected before deploy. |
| T-01-04 | Tampering | Azure CLI deploy without reviewing Bicep diff | mitigate | Task 3 checkpoint is blocking human-verify: developer reviews Bicep lint output before running deploy.sh. No ad-hoc az deployment group create. |
| T-01-SC | Tampering | Bicep references to external modules or registries | accept | postsapi.bicep references only built-in Azure resource types (Microsoft.Storage/*, Microsoft.Web/*, Microsoft.Authorization/*). No external module registry dependencies. |
</threat_model>
